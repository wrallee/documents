## 1. 한 페이지 요약

이 문서는 에이전틱 엔지니어링을 처음 접하는 숙련 백엔드 개발자가 **에이전트를 사용하는 사람**에서 **에이전트 시스템을 설계하고 운영하는 사람**으로 이동하기 위한 학습 로드맵입니다. 세부 실습 자료 자체가 아니라, 앞으로 다른 AI나 강사에게 실습 자료 생성을 요청할 수 있도록 **각 Step의 목표, 선행 조건, 입력 컨텍스트, 산출물, 완료 기준, 프롬프트 예시**를 명확히 정리한 커리큘럼 안내서입니다.

> **핵심 관점:** 에이전트는 인턴이고, 인간은 디렉터입니다. 에이전트에게 사고의 일부를 맡길 수는 있지만, 최종적인 이해·판단·품질 기준은 인간이 책임져야 합니다.

| Phase | 학습 초점 | 대표 산출물 | 이 단계에서 얻는 것 | 예상 기간 |
|---|---|---|---|---|
| Phase 0 | Antigravity CLI로 개념 체감 | Antigravity CLI 실행 환경, `/goal` 실행 기록, 하네스 관찰 메모 | 완성된 터미널 기반 코딩 에이전트를 써보며 컨텍스트, 스킬, 루프, 하네스의 직관을 얻습니다 | 2~3일 |
| Phase 1 | Pi coding agent로 조립하며 개념 익히기 | AGENTS.md, 리뷰 스킬, 승인 게이트 | 컨텍스트, 스킬, 하네스를 손으로 만져 보며 코드 리뷰 에이전트의 최소 구조를 이해합니다 | 1~2주 |
| Phase 2 | 관측성 및 루프 | Prometheus 메트릭, Grafana 대시보드, PR 기반 루프 | 에이전트가 어떻게 움직이는지 측정하고, 실패 원인을 추적할 수 있습니다 | 1~2주 |
| Phase 3 | Spring AI로 재구현 | 백엔드 서비스, Tool Calling, Memory, RAG, audit log, 팀 알림 | 실험용 에이전트를 운영 가능한 서비스 구조로 재구성합니다 | 2~4주 |
| Phase 4 | 나만의 에이전트 만들기 | 코딩 분신, 커리어 비서, 중국어 튜터 등 개인 프로젝트 | 앞선 개념을 활용해 진짜 원하는 에이전트를 설계하고 확장합니다 | 1~3주 이상 |

전체 흐름은 **체감 → 조립 → 관측 → 재구현 → 개인화**입니다. 처음부터 거대한 플랫폼을 만들지 않고, Antigravity CLI로 완성된 도구의 동작 방식을 체감한 뒤, Pi coding agent로 작동하는 경험을 만들고, 관측성과 루프를 붙이고, Spring AI 기반 서비스로 재구현합니다. 마지막에는 이 패턴을 자신의 실제 문제로 옮겨 **나만의 에이전트**를 만듭니다.

| 에이전틱 엔지니어링 4대 개념 | 이 커리큘럼에서의 의미 | 코드 리뷰 에이전트 예시 |
|---|---|---|
| **컨텍스트** | 에이전트가 작업을 이해하는 데 필요한 배경, 규칙, 제약, 코드베이스 지식입니다 | AGENTS.md에 리뷰 기준, 아키텍처 원칙, 금지 사항을 명시합니다 |
| **스킬** | 반복 가능한 작업 능력을 에이전트가 호출 가능한 형태로 정리한 것입니다 | 보안 리뷰, 테스트 관점 리뷰, 네이밍 리뷰 extension을 만듭니다 |
| **루프** | 이벤트를 감지하고 작업하고 피드백을 반영하는 반복 구조입니다 | PR 감지 → 리뷰 생성 → 사람 승인 대기 → 반영 흐름을 만듭니다 |
| **하네스** | 에이전트의 행동을 안전하게 제한하고 평가하며 승인하는 장치입니다 | 승인 게이트, audit log, 평가셋, 실행 제한을 붙입니다 |

## 2. 커리큘럼의 기본 철학

이 과정은 단순한 **Vibe Coding** 훈련이 아니라 **Agentic Engineering** 훈련입니다. Vibe Coding이 코드를 만드는 진입 장벽을 낮추고 개인 생산성의 바닥을 올리는 접근이라면, Agentic Engineering은 에이전트가 만든 결과물을 시스템적으로 검토하고, 관측하고, 승인하며, 품질 기준을 유지하는 접근입니다.

| 비교 항목 | Vibe Coding | Agentic Engineering |
|---|---|---|
| 주된 목적 | 빠르게 만들어 보기 | 지속적으로 품질을 지키며 만들기 |
| 인간의 역할 | 프롬프트를 주고 결과를 받는 사용자 | 기준을 설계하고 감독하는 디렉터 |
| 핵심 관심사 | 속도, 편의성, 아이디어 구현 | 컨텍스트 품질, 루프 안정성, 하네스, 관측성 |
| 실패 양상 | 작동은 하지만 이해하지 못하는 코드가 늘어납니다 | 과도한 자동화를 막고 검토 가능한 구조를 만듭니다 |
| 학습 목표 | 도구 숙련 | 에이전트 시스템 설계 능력 |

이 커리큘럼에서 인간 개발자는 에이전트에게 모든 것을 위임하지 않습니다. 오히려 **미학, 판단, 취향, 감독, 스펙 설계**를 책임지는 디렉터 역할을 수행합니다. 에이전트는 리뷰 초안을 만들고 반복 작업을 수행하지만, 최종 승인과 기준 수립은 개발자가 담당합니다.

> **운영 원칙:** 사고는 일부 외주화할 수 있지만, 이해는 외주화할 수 없습니다. 에이전트가 어떤 결론을 내렸는지, 그 결론이 왜 타당한지, 어떤 리스크가 남았는지는 인간이 설명할 수 있어야 합니다.

## 3. 전체 학습 로드맵

아래 로드맵은 “사용법 학습”에서 시작해 “운영 가능한 에이전트 시스템 설계”로 이동하고, 마지막에는 “나만의 문제를 해결하는 개인화 에이전트”로 확장됩니다. 각 Phase의 Step은 실습 자료 생성을 다른 AI에게 맡길 수 있도록 구조화되어 있으므로, 이 문서는 이후 실습 교안, 워크북, 과제 설명서, 사내 스터디 자료로 계속 진화할 수 있습니다.

| 순서 | 단계 | 핵심 질문 | 주요 학습 내용 | 완료 기준 |
|---:|---|---|---|---|
| 0 | Phase 0. 체감 | 완성된 에이전트 도구는 어떻게 움직이고 어디에서 위험해지는가? | Antigravity CLI, `/goal`, `/grill-me`, `/schedule`, 하네스 관찰 | 컨텍스트, 스킬, 루프, 하네스가 무엇인지 직관적으로 설명할 수 있습니다 |
| 1 | Phase 1. 조립 | 에이전트에게 무엇을 알려주고, 무엇을 시킬 것인가? | AGENTS.md, extension, 승인 게이트 | 사람이 승인하는 코드 리뷰 에이전트가 동작합니다 |
| 2 | Phase 2. 관측 | 에이전트가 잘하고 있는지 어떻게 알 것인가? | 메트릭, Grafana, Prometheus, 루프, 원인 추적 | PR 리뷰 흐름과 지표가 대시보드에서 보입니다 |
| 3 | Phase 3. 재구현 | 이 구조를 백엔드 서비스로 만들려면 무엇이 필요한가? | Spring AI, Tool Calling, Memory, RAG, audit log | 팀에서 사용할 수 있는 서비스 형태가 됩니다 |
| 4 | Phase 4. 개인화 | 내가 진짜 원하는 에이전트는 무엇인가? | 후보 선정, 컨텍스트 설계, 스킬 설계, 루프와 하네스 설계 | 개인 에이전트의 MVP와 확장 계획이 만들어집니다 |

## 4. Step 작성 규칙: 다른 AI에게 바로 넘길 수 있는 형태

이 문서의 각 Step은 단순한 할 일 목록이 아니라, 다른 AI에게 실습 자료를 생성하도록 요청할 수 있는 **메타 프롬프트 단위**입니다. 따라서 모든 Step은 동일한 항목을 포함합니다.

| 항목 | 의미 |
|---|---|
| 목표 | 이 Step에서 한 줄로 달성해야 하는 목적입니다 |
| 선행 조건 | 이전 Step 완료 여부, 필요한 계정, 도구, 코드 저장소, 환경을 뜻합니다 |
| 입력 컨텍스트 | 학습자 수준, 사용 도구, 제약사항, 실습 대상의 특징을 뜻합니다 |
| 대체 가능한 도구/기술 | Pi, Claude Code, Gemini, Spring AI 등 현재 도구가 바뀌어도 갈아끼울 수 있는 후보입니다 |
| 산출물 형태 | 실습이 끝났을 때 만들어져야 하는 구체적인 파일, 설정, 코드, 문서, 대시보드입니다 |
| 완료 기준 | 이 조건을 만족하면 해당 Step이 끝났다고 판단합니다 |
| AI 프롬프트 예시 | 다른 AI에게 이 Step의 실습 자료 생성을 요청할 때 사용할 수 있는 템플릿입니다 |

> **실습 자료 작성 원칙**:
> - 실습 자료는 학습자가 단계별로 따라갈 수 있는 흐름으로 구성한다.
> - 각 단계에서 필요한 터미널 명령어와 코드 snippet을 코드 블록으로 포함한다.
> - 소스 코드(Java, Python 등)는 해당 언어의 코드 블록으로만 제공하며, 터미널에서 파일을 생성하는 명령어(cat > file << EOF 등)로 감싸지 않는다. 파일 경로와 저장 위치는 텍스트로 안내한다.
> - 한 번에 모든 것을 복붙 실행하는 스크립트가 아니라, 학습자가 각 단계를 이해하며 따라갈 수 있는 설명 중심의 자료를 만든다.
> - AI 프롬프트 예시를 사용할 때는 반드시 이 작성 원칙을 함께 전달해야 합니다.

## 5. Phase 0: Antigravity CLI로 개념 체감

Phase 0의 목표는 완성된 터미널 기반 코딩 에이전트를 먼저 사용해 보면서, 이후 Phase 1에서 직접 조립할 **컨텍스트, 스킬, 루프, 하네스**의 의미를 감각적으로 이해하는 것입니다. Antigravity CLI는 Google이 2026년 5월 출시한 터미널 기반 코딩 에이전트로, 구 Gemini CLI를 Go로 재작성했으며 비동기 서브에이전트와 플러그인 시스템을 지원합니다.

이 단계에서는 직접 에이전트 프레임워크를 만들지 않습니다. 대신 `/goal`, `/grill-me`, `/schedule` 같은 핵심 명령을 사용해 에이전트가 목표를 분해하고, 스펙을 되묻고, 반복 실행을 구성하는 방식을 관찰합니다. 이를 통해 Phase 1에서 Pi coding agent로 직접 조립할 때 어떤 경계와 승인 지점이 필요한지 방향감을 잡습니다.

### 5.1 이 단계에서 얻는 것

| 획득 역량 | 설명 |
|---|---|
| 4대 개념의 직관적 이해 | 컨텍스트, 스킬, 루프, 하네스가 추상 개념이 아니라 실제 도구 동작의 일부임을 체감합니다 |
| 자율 실행 루프 감각 | `/goal` 실행을 통해 에이전트가 목표를 해석하고 여러 작업을 이어서 수행하는 흐름을 관찰합니다 |
| 컨텍스트 명확화 감각 | `/grill-me`를 통해 에이전트가 모호한 요구사항을 되묻고 스펙을 구체화하는 방식을 확인합니다 |
| 하네스의 필요성 | 에이전트가 멈춰야 할 곳에서 멈추지 않거나, 사람의 판단이 필요한 지점을 넘어가려는 상황을 관찰합니다 |
| Phase 1 방향감 | 완성된 도구의 동작을 기준으로, 직접 조립할 코드 리뷰 에이전트에 필요한 최소 구조를 이해합니다 |

### Step 1. Antigravity CLI 설치 및 Gemini API key 연결

| 항목 | 내용 |
|---|---|
| 목표 | Antigravity CLI를 설치하고 Gemini API key를 연결해 터미널에서 첫 실행이 가능한 환경을 만듭니다 |
| 선행 조건 | Google AI Studio에서 Gemini API key를 발급할 수 있어야 하며, 로컬 터미널과 실습용 Git 저장소가 준비되어 있어야 합니다 |
| 입력 컨텍스트 | 학습자는 백엔드 개발 경험이 있으며, 터미널 기반 코딩 에이전트를 사용해본 적이 있습니다. 이 단계의 목적은 도구 숙련 자체가 아니라 에이전틱 엔지니어링의 핵심 개념을 빠르게 체감하는 것입니다 |
| 대체 가능한 도구/기술 | Antigravity CLI 대신 유사한 터미널 기반 코딩 에이전트를 사용할 수 있지만, `/goal`, `/grill-me`, `/schedule`에 대응하는 자율 실행, 스펙 명확화, 반복 실행 기능이 있어야 합니다 |
| 산출물 형태 | 설치 기록, API key 연결 방식, 샘플 저장소에서 실행 가능한 기본 명령 목록, 초기 실행 확인 결과를 만듭니다 |
| 완료 기준 | Antigravity CLI가 샘플 저장소의 파일을 읽고, Gemini API key를 사용해 간단한 질의에 응답할 수 있으면 완료합니다 |

## 6. Phase 1: Pi coding agent로 조립하며 개념 익히기

Phase 1의 목표는 처음부터 완벽한 백엔드 서비스를 만드는 것이 아니라, Pi coding agent를 이용해 **작동하는 코드 리뷰 에이전트의 최소 구조**를 빠르게 조립하는 것입니다. 이 단계에서는 에이전트가 무엇을 알아야 하는지, 어떤 스킬을 가져야 하는지, 어디에서 사람의 승인을 받아야 하는지를 직접 경험합니다.

### 6.1 이 단계에서 얻는 것

| 획득 역량 | 설명 |
|---|---|
| 컨텍스트 설계 감각 | AGENTS.md에 어떤 정보를 넣어야 에이전트 결과물이 안정되는지 이해합니다 |
| 스킬 모듈화 감각 | 리뷰 기준을 extension 또는 재사용 가능한 작업 단위로 분리합니다 |
| 하네스의 필요성 | 에이전트가 곧바로 반영하지 않고, 인간 승인 이후에만 진행해야 하는 이유를 체감합니다 |
| 디렉터 역할 | 에이전트 결과를 판단하고 수정 지시를 내리는 기준을 세웁니다 |

## 7. Phase 2: 관측성 & 루프

Phase 2의 목표는 에이전트가 “돌아간다”는 사실을 넘어서, **어떻게 돌아가고 있는지 측정할 수 있는 상태**를 만드는 것입니다. Prometheus는 메트릭을 수집하고 저장하는 오픈소스 모니터링 시스템이며, Grafana는 수집된 지표를 대시보드로 시각화하는 데 널리 사용됩니다.[2] [3] 이 단계에서는 코드 리뷰 에이전트의 PR 감지, 리뷰 실행, 승인 대기, 반영 흐름을 루프로 설계하고, 이상 징후가 발생했을 때 원인을 추적할 수 있도록 만듭니다.

### 7.1 이 단계에서 얻는 것

| 획득 역량 | 설명 |
|---|---|
| 운영 관점 | 에이전트를 일회성 스크립트가 아니라 관측 가능한 시스템으로 바라봅니다 |
| 루프 설계 능력 | PR 감지부터 승인 대기까지의 상태 전이를 설계합니다 |
| 메트릭 설계 능력 | 리뷰 소요 시간, 승인율, 재작업률, 실패율 같은 지표를 정의합니다 |
| 원인 추적 능력 | 이상 징후가 발생했을 때 컨텍스트, 스킬, 모델 응답, 외부 도구 호출 중 어디가 문제인지 좁혀 갑니다 |

| 대체 가능한 도구/기술 | Prometheus metric naming, OpenTelemetry metric, StatsD, CloudWatch Metrics, Datadog metric을 사용할 수 있습니다 |
| 산출물 형태 | 메트릭 이름, 타입, 설명, 라벨, 수집 위치가 포함된 메트릭 설계표를 만듭니다 |
| 완료 기준 | 에이전트의 처리량, 실패율, 지연, 승인 대기, 재작업을 설명하는 핵심 메트릭 목록이 정리되면 완료합니다 |
| AI 프롬프트 예시 | “코드 리뷰 에이전트의 동작 메트릭을 정의하는 실습 자료를 만들어 주세요. 리뷰 요청 수, 성공률, 실패율, 리뷰 지연 시간, 승인 대기 시간, 재작업 횟수, 컨텍스트 버전을 포함하고, Prometheus 스타일의 이름과 라벨 설계를 제안해 주세요.” |

## 8. Phase 3: Spring AI로 재구현

Phase 3의 목표는 앞선 두 단계에서 만든 실험적 구조를 **백엔드 서비스 아키텍처**로 재구현하는 것입니다. Spring AI는 Spring 애플리케이션에서 모델 호출, Tool Calling, RAG, 벡터 저장소 연동 같은 AI 애플리케이션 구성 요소를 다루는 프레임워크입니다.[4] 이 단계는 백엔드 개발자의 강점을 적극 활용합니다. 즉, 도구 사용자가 아니라 서비스 설계자로서 에이전트의 경계, 상태, 권한, 감사 가능성, 배포 방식을 정의합니다.

### 8.1 이 단계에서 얻는 것

| 획득 역량 | 설명 |
|---|---|
| 서비스화 능력 | 에이전트 흐름을 API, job, queue, storage, notification으로 분해합니다 |
| Spring AI 적용 능력 | Tool Calling, Memory, RAG를 코드 리뷰 도메인에 연결합니다 |
| 하네스 고도화 | audit log, 평가셋, 승인 게이트를 통해 운영 리스크를 줄입니다 |
| 팀 배포 감각 | Slack 또는 DingTalk을 통해 사람이 개입할 수 있는 협업 흐름을 만듭니다 |

| AI 프롬프트 예시 | “코드 리뷰 에이전트를 Spring 기반 백엔드 서비스로 재구현하기 위한 서비스 경계 설계 실습 자료를 만들어 주세요. Review API, Agent Orchestrator, Tool Layer, Memory Store, RAG Store, Approval Gate, Notification Adapter, Observability의 책임과 데이터 흐름을 포함해 주세요.” |

### 8.2 서비스 아키텍처 관점

## 9. Phase 4: 나만의 에이전트 만들기

Phase 4의 목표는 Phase 1~3에서 익힌 **컨텍스트, 스킬, 루프, 하네스** 개념을 활용해 학습자가 진짜 원하는 에이전트를 직접 설계하는 것입니다. 현재 후보는 아래 3가지로 고정되어 있으며, 그중 **중국어 튜터 에이전트**가 가장 우선순위가 높습니다.

### 9.1 후보 프로젝트 목록

| 후보 | 설명 | 핵심 컨텍스트 | 주요 스킬 | 루프와 하네스 |
|---|---|---|---|---|
| 코딩 분신 에이전트 | 사용자의 코드 스타일과 설계 취향을 반영해 구현 초안을 작성합니다 | 과거 코드, 선호 패턴, 금지 패턴 | 기능 구현, 리팩터링, 테스트 작성 | 사용자가 diff를 검토하고 승인합니다 |
| 커리어 성장 비서 | 학습 목표, 이력, 프로젝트 경험을 바탕으로 성장 계획을 제안합니다 | 경력 목표, 현재 역량, 관심 도메인 | 학습 계획, 회고 질문, 포트폴리오 정리 | 월간 회고와 목표 재조정을 둡니다 |
| 중국어 튜터 에이전트 | 간체자 읽기, 말하기, 병음 입력을 훈련합니다 | 현재 수준, 관심 주제, 오답 이력 | 문장 읽기, 발음 피드백, 병음 퀴즈 | 오답 복습 루프와 난이도 조절을 둡니다 |

### 9.2 이 단계에서 얻는 것

| 획득 역량 | 설명 |
|---|---|
| 문제 선정 능력 | 에이전트가 풀 만한 문제와 풀면 안 되는 문제를 구분합니다 |
| 패턴 전이 능력 | 코드 리뷰 에이전트에서 배운 구조를 다른 도메인에 적용합니다 |
| 개인화 설계 능력 | 자신의 취향, 목표, 학습 이력, 작업 스타일을 컨텍스트로 정리합니다 |
| 지속 개선 능력 | 새 도구와 새 요구가 생길 때 에이전트를 진화시키는 방법을 익힙니다 |

## 14. 업데이트 가이드

이 문서는 고정된 교재가 아니라 **진화하는 가이드라인**입니다. 에이전틱 엔지니어링 도구와 프레임워크는 빠르게 바뀌므로, 특정 도구 이름보다 컨텍스트, 스킬, 루프, 하네스라는 구조를 유지하는 것이 중요합니다. 새 도구가 나오거나 새 프로젝트 후보가 생기면 아래 규칙에 따라 문서를 업데이트합니다.

### 14.1 새 도구가 나왔을 때

| 업데이트 지점 | 수정 방법 |
|---|---|
| Step의 대체 가능한 도구/기술 | 기존 도구를 삭제하기보다 새 도구를 대체 후보로 추가하고, 어떤 조건에서 갈아끼울지 설명합니다 |
| 환경 및 준비물 | 새 도구가 필수 도구가 아니라면 “대체 가능성” 칸에 먼저 추가합니다 |
| 산출물 형태 | 새 도구가 산출물의 형식을 바꾸면 기존 산출물과의 대응 관계를 함께 적습니다 |
| 완료 기준 | 도구별 명령 성공 여부보다 학습 목표 달성 여부를 기준으로 유지합니다 |

### 14.2 새 프로젝트 후보가 생겼을 때

| 업데이트 지점 | 수정 방법 |
|---|---|
| Phase 4 후보 프로젝트 목록 | 새 후보를 한 줄 추가하고, 핵심 컨텍스트, 주요 스킬, 루프와 하네스를 함께 적습니다 |
| Phase 4 Step 1 | 후보 비교표에 새 후보를 포함하도록 프롬프트 예시를 갱신합니다 |
| Phase 4 Step 2~5 | 새 후보에 맞는 컨텍스트, 스킬, 루프, MVP 예시를 추가합니다 |
| 학습 완료 기준 | 새 후보가 기존 완료 기준을 바꾸는지 확인하되, 가능하면 공통 기준을 유지합니다 |

### 14.3 새 운영 요구가 생겼을 때

| 운영 요구 | 문서 반영 방법 |
|---|---|
| 보안 요구가 강화됨 | 하네스, 권한 제한, audit log, 개인정보 취급 원칙을 갱신합니다 |
| 팀 배포 채널이 바뀜 | Slack 또는 DingTalk 항목을 유지하되 대체 채널을 추가합니다 |
| 모델 provider가 바뀜 | 모델 이름보다 provider abstraction과 설정 방식 중심으로 수정합니다 |
| 관측성 스택이 바뀜 | Prometheus와 Grafana를 기준점으로 유지하되 OpenTelemetry, Datadog, New Relic 등으로 대체하는 경로를 적습니다 |
| 평가 방식이 정교해짐 | 평가셋, golden set, LLM-as-judge, human review rubric을 추가합니다 |

### 14.4 버전 관리 규칙

| 변경 유형 | 버전 증가 기준 | 예시 |
|---|---|---|
| Patch | 오탈자, 표현, 링크 수정 | v1.1.0 → v1.1.1 |
| Minor | Step 추가, 대체 도구 추가, 후보 프로젝트 추가 | v1.1.0 → v1.2.0 |
| Major | Phase 구조 변경, 핵심 철학 변경, 대상 독자 변경 | v1.1.0 → v2.0.0 |

문서를 업데이트할 때는 상단의 **버전, 최종 수정일, 변경 이력**을 반드시 함께 수정합니다. 각 Step의 프롬프트 예시도 도구와 환경 변화에 맞춰 갱신해야 합니다. 특히 특정 도구가 사라지거나 다른 도구가 주류가 되더라도, Step의 목표와 완료 기준은 가능하면 그대로 유지해 학습 구조가 흔들리지 않게 합니다.

## 15. 마무리

이 커리큘럼의 핵심은 에이전트를 “코드를 대신 짜는 도구”로만 보지 않는 것입니다. 에이전트는 컨텍스트를 먹고, 스킬을 사용하며, 루프 안에서 움직이고, 하네스에 의해 안전하게 제한되는 **소프트웨어 시스템의 일부**입니다. 따라서 에이전틱 엔지니어링을 배운다는 것은 모델을 잘 부르는 법을 배우는 것을 넘어, **인간의 판단과 시스템의 자동화가 만나는 경계**를 설계하는 법을 배우는 것입니다.

최종적으로 학습자는 Antigravity CLI로 완성된 에이전트 도구의 동작을 먼저 체감한 뒤, 코드 리뷰 에이전트를 통해 에이전틱 엔지니어링의 기본기를 익히고, 관측성과 자기 개선 루프를 붙이며, Spring AI 기반 서비스로 재구현할 수 있게 됩니다. 이어서 코딩 분신, 커리어 성장 비서, 중국어 튜터처럼 자신이 실제로 원하는 에이전트를 직접 만들며, 이 문서를 계속 업데이트 가능한 학습 가이드로 발전시킬 수 있습니다.
